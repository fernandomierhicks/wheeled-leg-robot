## Controller Details

### 1. Balance LQR (inner, 500 Hz)

**State vector:**
```
x = [pitch вҲ’ Оё_ref, pitch_rate, wheel_vel_avg вҲ’ v_ref]   вҶҗ 3-state
```

`wheel_pos` is NOT in the state. Position hold in BALANCE mode is the outer loop's job (Оё_ref from VelocityPI). This keeps the LQR compatible with intentional forward motion in DRIVE mode.

**LQR formulation:**
```
u = вҲ’K @ x
K = [K_pitch, K_pitch_rate, K_vel]
Q = diag([Q_pitch, Q_pitch_rate, Q_vel])
R = scalar
```

**Important вҖ” `v_ref` coupling:** The LQR third state is `(wheel_vel вҲ’ v_ref)`, where `v_ref = v_desired / WHEEL_R`. This means the LQR does not penalise wheel velocity when the robot is intentionally moving at the commanded speed. Both the headless simulation and the replay must pass `v_ref = v_desired / WHEEL_R` вҖ” passing `v_ref=0.0` during drive incorrectly penalises the wheel velocity and fights the PI's lean command.

**Optimized gains (scenario 4_leg_height_gain_sched, 5-min / 1896 evals, 2026-03-18) вҖ” CURRENT BASELINE:**
| Parameter | Value | Notes |
|-----------|-------|-------|
| Q_PITCH | 0.014168 | soft pitch weight вҖ” ~10Г— lower than S1 |
| Q_PITCH_RATE | 0.033720 | dominant damping term вҖ” slightly higher than S1 |
| Q_VEL | 0.000250 | minimal velocity correction вҖ” ~18Г— lower than S1 |
| R | 28.734420 | moderate torque cost вҖ” ~3Г— higher than S1 (softer) |
| fitness | 0.017938 | RMS pitch 1.242В°, survived 12 s (3 full leg cycles) |

**Prior S1 gains (retained for reference вҖ” do not use as baseline):**
| Parameter | Value | Scenario |
|-----------|-------|---------|
| Q_PITCH | 0.138282 | `1_LQR_pitch_step`, 5-min / 4448 evals, fitness=0.003267 |
| Q_PITCH_RATE | 0.023379 | RMS pitch 0.926В°, survived 5 s |
| Q_VEL | 0.004591 | вҖ” |
| R | 9.998298 | вҖ” |

**Character of the S4 gains:** Softer overall than S1 вҖ” lower Q_PITCH, higher R. The optimizer
settled on a solution that tolerates more pitch excursion in exchange for lower torque, which is
sensible for a 12s scenario where the leg-induced equilibrium shifts are the primary stimulus.
Q_PITCH_RATE is slightly higher than S1, preserving damping. Steady-state pitch offset is **not
corrected by the LQR** вҖ” that is intentional and handled by the outer Velocity PI loop (see
В§"Why We Do NOT Use LQI" below).

**Gain Scheduling (Ascento-style):**

The 4-bar stroke (~62В°) changes effective pendulum length l_eff by ~50 mm.
Gains are precomputed at 3 operating points and interpolated online:

```python
K_table = {
    'retracted': compute_lqr_gain(q_hip=Q_RET),   # legs short
    'nominal':   compute_lqr_gain(q_hip=Q_NOM),   # default stance
    'extended':  compute_lqr_gain(q_hip=Q_EXT),   # legs long
}

# Online at 500 Hz:
alpha = (q_hip - Q_RET) / (Q_EXT - Q_RET)
K = (1 - alpha) * K_table['retracted'] + alpha * K_table['extended']
```

l_eff is computed via `physics.get_equilibrium_pitch(p, q_hip)` at each table entry.

### 2. Velocity PI (outer, 500 Hz)

Converts velocity command into a lean angle setpoint for the LQR:

```python
vel_est_ms = wheel_vel * WHEEL_R          # +wheel_vel = forward body motion
v_err = v_desired вҲ’ vel_est_ms
Оё_ref = Kp_v * v_err + Ki_v * integral(v_err)
Оё_ref = clamp(Оё_ref, вҲ’0.26 rad, +0.26 rad)   # Вұ15В° hard limit
```

**BALANCE mode:** `v_desired = 0` вҶ’ PI integrates any drift into a restoring lean.
**DRIVE mode:** `v_desired = slider / joystick command` вҶ’ PI drives to target speed.

**Sign convention:** positive `wheel_vel` = forward body motion = positive `vel_est_ms`.
Positive `Оё_ref` = lean forward = drive forward.

**Integrator reset:** On direction reversal (sign of `v_desired` flips), the PI integrator is
reset to zero to prevent windup carryover. Implemented in `_run_sim_loop` and replay.

**Optimized gains (S5 `5_VEL_PI_leg_cycling`, 5-min / 1776 evals, 2026-03-18) вҖ” CURRENT BASELINE:**
| Parameter | Value | Notes |
|-----------|-------|-------|
| Kp_v | 0.251209 | rad/(m/s) вҖ” ~ВҪ of prior combined_PI value; softer to absorb bump hits |
| Ki_v | 0.011405 | rad/m вҖ” similar to prior |
| fitness | 2.0635 | S5: vel_rms=0.502 m/s, pitch=5.59В°, survived 13 s |
| THETA_REF_RATE_LIMIT | 2.0 rad/s | rate-limits lean command to suppress pitch-rate spikes on step changes |

**Prior combined_PI baseline (retained for reference):**
| Parameter | Value | Scenario |
|-----------|-------|---------|
| Kp_v | 0.502932 | combined_PI, 30-min / 5488 evals, fitness=0.61 |
| Ki_v | 0.012678 | вҖ” |

**Flag:** `USE_VELOCITY_PI` in `scenarios.py` вҖ” set `False` to disable outer loop for LQR-only tests.

### 3. Yaw PI (outer, 500 Hz)

Produces differential torque for turning. Blends with drive.

```python
yaw_err = Пү_yaw_desired вҲ’ Пү_yaw_measured   # from BNO086 gyro Z
П„_yaw = Kp_yaw * yaw_err + Ki_yaw * integral(yaw_err)
П„_yaw = clamp(П„_yaw, вҲ’0.5 NВ·m, +0.5 NВ·m)

П„_L = П„_sym + П„_yaw
П„_R = П„_sym вҲ’ П„_yaw
```

Starting gains: Kp_yaw = 0.3, Ki_yaw = 0.05. **Not yet implemented.**

### 4. Leg Impedance PD + Roll Leveling (independent, 500 Hz)

Decoupled from balance. Per-leg spring-damper with differential roll leveling:

```python
# Roll leveling вҖ” differential hip target
roll_true = atan2(2*(w*x + y*z), 1 - 2*(xВІ+yВІ))   # from free joint quaternion
roll_meas = roll_true + N(0, ROLL_NOISE_STD_RAD)     # BNO086 noise (0.05В°)
roll_rate = data.qvel[d_root + 3]                    # Пүx world-frame

Оҙq = K_ROLL * roll_meas + D_ROLL * roll_rate
q_nom_L = clamp(Q_NOM + Оҙq, HIP_SAFE_MIN, HIP_SAFE_MAX)
q_nom_R = clamp(Q_NOM - Оҙq, HIP_SAFE_MIN, HIP_SAFE_MAX)

# Spring-damper per leg
П„_hip = clamp(-(K_s*(q_hip - q_nom) + B_s*dq_hip), ВұHIP_IMPEDANCE_TORQUE_LIMIT)
```

**Baselined gains (Phase 4 optimizer, S5+S8 combined, 177 gens / 1416 evals, 2026-03-18):**
| Param | Value | Notes |
|-------|-------|-------|
| K_s | 16.000000 NВ·m/rad | Very stiff spring |
| B_s | 0.821633 NВ·mВ·s/rad | Low damping вҖ” spring-dominant |
| K_ROLL | 3.963615 rad/rad | Aggressive roll correction |
| D_ROLL | 1.000000 radВ·s/rad | Strong roll rate damping |
| HIP_IMPEDANCE_TORQUE_LIMIT | 1.0 NВ·m | Keeps hips backdrivable |
| HIP_SAFE_MIN | Q_EXT + 10В° = вҲ’1.257 rad | Buffer from extended limit |
| HIP_SAFE_MAX | Q_RET вҲ’ 10В° = вҲ’0.526 rad | Buffer from retracted limit |

Roll leveling is **orthogonal** to pitch/yaw вҖ” Оҙq has zero effect on average hip height.
Robot drives at Q_NOM (30% of stroke) to maximise suspension travel in both directions.

## Optimizer Scenarios

All scenarios run at 500 Hz controller rate (2 kHz physics, CTRL_STEPS=4). Noise model active by default (BNO086 pitch/rate noise). All scenarios use `_run_sim_loop` as the shared simulation backend вҖ” S2 and S3 run byte-for-byte identical physics.

### Scenario 1 вҖ” `1_LQR_pitch_step` *(superseded by S4 for LQR tuning)*

**Purpose:** Isolate and tune the Balance LQR inner loop. VelocityPI is OFF. Legs held at Q_NOM вҖ” leg height variation is not tested.

**Initial condition:** Equilibrium pitch + 5В° perturbation (PITCH_STEP_RAD).

**Controller under test:** Balance LQR only. `Оё_ref = 0`, `v_ref = 0`.

**Disturbances:**
- +4N at t=2.0s for 0.2s (forward kick)
- вҲ’4N at t=3.0s for 0.2s (backward kick)

**Duration:** 5s

**Metric / Fitness:** ISE (integral of squared pitch error) in radВІВ·s вҖ” lower is better.
```
fitness = W_FALL Г— fell  +  ОЈ (pitch_errВІ) Г— dt
```

**What is being optimised:** Q_PITCH, Q_PITCH_RATE, Q_VEL, R вҖ” the LQR cost weights.

**Pass criterion:** Survives full 5s, RMS pitch < 2В°, no wheel liftoff.

**Optimizer:** `optimize_lqr.py` вҖ” (1+8)-ES over the 4 Q/R parameters.

**Last best:** Q=[0.138, 0.0234, 0.00459], R=9.998, fitness=0.003267 (4448 evals, 2026-03-18).

> **Note:** S4 (`4_leg_height_gain_sched`) supersedes S1 for LQR Q/R optimisation.
> S4 includes the same disturbances but additionally exercises the gain scheduler across
> the full leg stroke вҖ” gains tuned on S4 are strictly more general than S1-tuned gains.
> `optimize_lqr.py` is now pointed at S4.

### Scenario 4 вҖ” `4_leg_height_gain_sched` *(current LQR optimizer target)*

**Purpose:** Tune the Balance LQR Q/R weights while the gain scheduler is fully exercised. The legs cycle sinusoidally through the operating range, changing the effective pendulum length continuously. This makes S4 a strict superset of S1: any Q/R set that handles S4 also handles fixed-height balance.

**Initial condition:** Equilibrium pitch at Q_NOM. No pitch perturbation вҖ” the leg motion itself is the stimulus.

**Controller under test:** Balance LQR only (gain-scheduled). `Оё_ref = 0`, `v_ref = 0`. VelocityPI OFF.

**Leg cycle:**
```
q_hip(t) = center + amp Г— sin(2ПҖ t / 4.0)

LEG_CYCLE_Q_RET = Q_RET вҲ’ 5В°  вүҲ вҲ’25.1В°   (less crouched than full-retract to avoid 4-bar instability)
Q_EXT           = вҲ’82.0В°                   (fully extended)
center          вүҲ вҲ’53.6В°,  amp вүҲ вҲ’28.5В°
Period = 4 s  вҶ’  3 full cycles in 12 s
```

**Disturbances** (same as S1):
- +4N at t=2.0s for 0.2s (fires mid first cycle, legs near mid-stroke)
- вҲ’4N at t=3.0s for 0.2s

**Duration:** 12s (3 full leg cycles)

**Metric / Fitness:**
```
fitness = ISE_pitch  +  W_PITCH_RATE Г— ISE_pitch_rate  +  W_FALL Г— fell
```
Pitch error is measured against the **live** `pitch_ff(q_hip)` equilibrium, not a fixed Q_NOM reference вҖ” so the score purely reflects how well the LQR tracks the changing equilibrium as leg height varies.

**What is being optimised:** Q_PITCH, Q_PITCH_RATE, Q_VEL, R вҖ” the LQR cost weights.

**Optimizer:** `optimize_lqr.py` вҖ” (1+8)-ES, seeded from S1 best.

**Seed (from S1):** Q=[0.138282, 0.023379, 0.004591], R=9.998298.

**Status:** вң… Baselined 2026-03-18 вҖ” 237 gens, 1896 evals, 5 min. Best: Q=[0.014168, 0.033720, 0.000250], R=28.734, fitness=0.017938, RMS pitch 1.242В°, survived 12 s.

### Scenario 2 вҖ” `2_VEL_PI_disturbance`

**Purpose:** Tune the Velocity PI outer loop for position hold quality under external disturbances. The LQR Q/R are fixed at their S1-optimised values.

**Initial condition:** Equilibrium pitch, no perturbation.

**Controller under test:** VelocityPI (outer) + Balance LQR (inner). `v_desired = 0` throughout вҖ” position hold only.

**Disturbances:**
- +1N at t=2.0s for 0.2s (lighter than S1 вҖ” tests PI hold, not LQR recovery)
- вҲ’1N at t=3.0s for 0.2s

**Duration:** 6s (gives PI 2.8s to correct after the last kick at t=3.2s)

**Metric / Fitness:**
```
fitness = (wheel_travel_m + W_FALL Г— fell) / duration   [units: m/s]
```
Absolute wheel travel measures how far the robot drifts вҖ” lower = better position hold.
Normalised by duration so it is on the same scale as S3's `vel_track_rms_ms`.

**What is being optimised:** KP_V, KI_V вҖ” the VelocityPI gains.

**Pass criterion:** Survives 6s with < 2m total wheel travel.

**Optimizer:** `optimize_vel_pi.py` вҖ” (1+8)-ES over KP_V/KI_V.

**Current best:** KP_V=0.503, KI_V=0.0127, fitness=0.61 (combined_PI, 5488 evals, 2026-03-18).

### Scenario 3 вҖ” `3_VEL_PI_staircase`

**Purpose:** Tune the Velocity PI for velocity tracking across a four-step speed profile including direction reversal. Complements S2 (which tests position hold) вҖ” together they ensure the PI works in both static and dynamic modes.

**Initial condition:** Equilibrium pitch, no perturbation.

**Controller under test:** VelocityPI (outer) + Balance LQR (inner).

**Disturbances:** None.

**Velocity profile:**
```
t =  0.0 вҖ“  1.0 s  вҶ’  0.0 m/s  (settle; excluded from VEL_ERR metric)
t =  1.0 вҖ“  3.0 s  вҶ’ +0.3 m/s  gentle forward
t =  3.0 вҖ“  5.0 s  вҶ’ +0.6 m/s  step up
t =  5.0 вҖ“  7.0 s  вҶ’ +1.0 m/s  high-speed forward
t =  7.0 вҖ“  9.0 s  вҶ’ вҲ’0.5 m/s  direction reversal вҖ” integrator reset fires
t =  9.0 вҖ“ 11.0 s  вҶ’ вҲ’1.0 m/s  high-speed backward
t = 11.0 вҖ“ 13.0 s  вҶ’  0.0 m/s  return to stop
```

**Duration:** 13s

**Metric / Fitness:**
```
fitness = W_VEL_ERR Г— vel_track_rms_ms  +  0.1 Г— W_RMS Г— rms_pitch_deg  +  W_FALL Г— fell
```
`vel_track_rms_ms` = RMS of `(v_commanded вҲ’ v_measured)` from t=1.0s onward, in m/s.
Pitch penalty is secondary (0.1Г—) вҖ” exists to prevent the PI from solving tracking by leaning aggressively.

**What is being optimised:** KP_V, KI_V вҖ” the VelocityPI gains (shared with S2).

**Pass criterion:** vel_track_rms_ms < 0.15 m/s, survives full 8.5s.

**Optimizer:** `optimize_vel_pi.py` with `combined_PI` scenario (0.5 Г— S2 + 0.5 Г— S3) to jointly optimise for both position hold and velocity tracking.

**Status:** вң… Baselined 2026-03-18 via combined_PI (5488 evals, 30 min).

### Scenario 5 вҖ” `5_VEL_PI_leg_cycling` *(current VelocityPI optimizer target)*

**Purpose:** Tune the Velocity PI under the hardest realistic conditions: legs cycling continuously while the robot simultaneously tracks a velocity staircase and absorbs disturbance kicks. S5 is a strict superset of both S2 (disturbances) and S3 (staircase) вҖ” gains that work here are the most transfer-robust.

**Initial condition:** Equilibrium pitch at Q_NOM, `v_desired = 0`.

**Controller under test:** VelocityPI (outer) + Balance LQR (inner, gain-scheduled). LQR Q/R fixed at S4 baseline.

**Velocity profile:** Identical to S3 staircase:
```
t =  0.0 вҖ“  1.0 s  вҶ’  0.0 m/s  (settle)
t =  1.0 вҖ“  3.0 s  вҶ’ +0.3 m/s  gentle forward
t =  3.0 вҖ“  5.0 s  вҶ’ +0.6 m/s  step up
t =  5.0 вҖ“  7.0 s  вҶ’ +1.0 m/s  high-speed forward
t =  7.0 вҖ“  9.0 s  вҶ’ вҲ’0.5 m/s  direction reversal
t =  9.0 вҖ“ 11.0 s  вҶ’ вҲ’1.0 m/s  high-speed backward
t = 11.0 вҖ“ 13.0 s  вҶ’  0.0 m/s  return to stop
```

**Disturbances:** S2 kicks: +1N at t=2s for 0.2s, вҲ’1N at t=3s for 0.2s (fire during the +0.3m/s phase, adding velocity AND pitch disturbance simultaneously).

**Leg motion:** Sinusoidal Q_RET вҶ” Q_EXT, period = 4s (same as S4) вҖ” 3+ full cycles during the 13s run. Changing leg height continuously shifts pitch_ff, challenging the VelocityPI + LQR cascade throughout.

**Duration:** 13s

**Metric / Fitness:**
```
fitness = W_VEL_ERR Г— vel_track_rms_ms  +  0.1 Г— W_RMS Г— rms_pitch_deg  +  W_FALL Г— fell
```
Same form as S3 вҖ” velocity tracking is primary, pitch penalty is secondary.

**What is being optimised:** KP_V, KI_V вҖ” the VelocityPI gains.

**Optimizer:** `optimize_vel_pi.py` (ACTIVE_SCENARIO updated to S5).

**Status:** вң… Baselined 2026-03-18 вҖ” 222 gens, 1776 evals, 5 min. Best: KP_V=0.251209, KI_V=0.011405, fitness=2.0635, vel_rms=0.502 m/s, pitch=5.59В°, survived 13 s.

### Combined PI Scenario вҖ” `combined_PI`

Used by the optimizer to jointly tune KP_V/KI_V for both behaviours:

```
fitness = 0.5 Г— S2_fitness  +  0.5 Г— S3_fitness
```

Both sub-fitnesses are normalised to m/s units before weighting. This prevents either scenario from dominating and ensures the PI works well across position hold and velocity tracking.

## Shared Simulation Infrastructure

All scenarios from S2 onwards use `_run_sim_loop` in `scenarios.py` as the single shared physics backend. This guarantees:

- Identical PI controller instantiation (same `VelocityPI` class, same gains)
- Correct `v_ref = v_target_ms / WHEEL_R` passed to the LQR at every step
- Consistent PI integrator reset on direction reversal
- Identical noise model, sensor read order, and leg impedance controller
- Same liftoff detection and settle-time tracking

The `dist_fn(t) -> float` callable interface allows each scenario to specify an arbitrary disturbance profile without duplicating the sim loop.

## Why We Do NOT Use LQI (LQR + Integrator on Pitch)

### Ascento precedent
Ascento (ETH Zurich, ICRA 2019) вҖ” the reference robot for this architecture вҖ” uses pure LQR for balance without integral action on pitch. The outer Velocity PI loop provides all necessary steady-state correction.

### The Velocity PI already IS the integrator
In BALANCE mode (`v_desired = 0`), the Velocity PI integrates wheel velocity error:
```
вҲ« v_err dt  вҶ’  Оё_ref  вҶ’  LQR lean command
```
Any persistent pitch offset that causes the robot to drift forward is continuously integrated by the PI and corrected via the Оё_ref lean command. This is **indirect integral action on position** вҖ” it achieves zero steady-state drift without touching the LQR.

### Why LQI would conflict
Adding a pitch integrator inside the LQR creates **two competing integrators** in the same loop:

```
LQI pitch integrator  вҶ’  wheel torque  вҶ’  velocity drift
Velocity PI integrator вҶ’  Оё_ref        вҶ’  wheel torque
```

Both try to null pitch error simultaneously via different paths. This causes integrator wind-up interactions, potential instability, and makes tuning significantly harder.

### Why Scenario 1 shows steady-state offset
Scenario 1 runs with `USE_VELOCITY_PI = False` вҖ” the outer integrator is deliberately disabled to isolate LQR tuning. The residual ~0.74В° pitch offset is **expected and correct** for a pure proportional LQR with no outer loop. It is NOT a sign the architecture is wrong.

### Conclusion
- **Do not add LQI.** The architecture is correct.
- The steady-state error visible in Scenario 1 headless tests is a testing artifact.
- The combined test (LQR + Velocity PI active) is the real measure of steady-state performance.
